## 2) M1e failure recap

- `build/StriV.Engine.M1e.slnf` includes `Stride.Engine` but not `Stride.Shaders.Compiler` directly; shader/compiler projects are therefore transitive from project references, not explicitly listed in the filter.  
- `Stride.Engine.csproj` directly references `Stride.Shaders.Compiler.csproj`, so admitting `Stride.Engine` necessarily admits shader compiler stack in current graph.  
- The reported first blocker (`CS0246` for `CppNet`) aligns with `PreProcessor.cs` importing `CppNet` namespace and instantiating `CppNet.Preprocessor`.
- Focus on shader/compiler/parser is justified because this is the first hard compile blocker before later VR/audio/native failures.

## 3) `CppNet` provenance audit

### Direct evidence
- `Stride.Core.Shaders.csproj` has an explicit assembly reference: `<Reference Include="CppNet">` with hint path `../../../deps/CppNet/netstandard1.3/CppNet.dll`.  
- `deps/CppNet` exists and contains `netstandard1.3/CppNet.dll` + `.pdb`, plus `checkout.bat` and `build.bat` scripts.
- `deps/CppNet/checkout.bat` clones `stride3d/CppNet` into `externals/CppNet`; `build.bat` builds via MSBuild and copies output dll into `deps/CppNet/netstandard1.3`.
- No `PackageReference Include="CppNet"` is present in `Directory.Packages.props` (or examined csproj files), and no `CppNet.csproj` exists in-repo.

### Conclusion
- `CppNet` is a **vendored binary dependency** (dll/pdb under `deps`), produced by an external source repo and copied in by legacy scripts; it is **not** a NuGet package in this tree, not an in-repo source project, and not generated by current SDK targets.
- Missing `CppNet` in M1e is most likely resolution/copy/reference mismatch (e.g., hint-path not valid at compile time in that environment/TFM graph), not absence of source usage.
- Provenance appears legacy/manual and partially Windows-centric (batch scripts + MSBuild workflow), though the consumed artifact target folder is `netstandard1.3`.

## 4) `Stride.Core.Shaders` audit

- Owner project of `PreProcessor.cs`: `sources/shaders/Stride.Core.Shaders/Stride.Core.Shaders.csproj` (default SDK compile include; no conditional exclusion of parser files observed).
- Direct references:
  - Assembly reference to `CppNet` via `HintPath`.
  - Project references to `Stride.Core` and `Irony`.
- Assembly processor/runtime flags are enabled (`StrideRuntime=true`, `StrideAssemblyProcessor=true`, serialization option).
- `CppNet` is additionally included in package output via custom target (`BuildOutputInPackage Include="$(TargetDir)CppNet.dll"`).
- This project mixes broad shader AST/parser infrastructure + preprocessing, suggesting **compiler/tooling-core plus shared shader model types**, not purely runtime.

### Feasibility judgment
- In current form, Linux feasibility depends on reliable availability of the vendored `CppNet.dll` artifact for all admitted target combinations.
- For Stri‑V Core separation goals, this project is a poor boundary because it bundles both reusable shader model objects and source-preprocessing implementation.

Recommendation: defer deep rewrite in M1e; prefer narrow opt-out path that avoids requiring this compile path from `Stride.Engine` first.

## 5) `PreProcessor.cs` deep inspection

- Namespace/type: `Stride.Core.Shaders.PreProcessor` partial class.
- CppNet APIs used directly: `Preprocessor`, `Feature`, `Warning`, `StringLexerSource`, `Token`, `PreprocessorListenerBase`, `Source`.
- Behavior implemented:
  - configures C-preprocessor features/warnings,
  - injects macro defines,
  - configures quote include paths,
  - token-stream loop with comment handling,
  - throws `LexerException` on preprocessor errors.
- Usage profile (static inference): parser/compiler pipeline concern (source preprocessing) rather than runtime effect bytecode consumption.

### Stub viability
- A no-op stub (e.g., return input source) is technically plausible for compile-only progression.
- Behavioral loss: macro expansion, includes, conditional compilation, diagnostics; high risk of silent shader semantic breakage.
- Risk for M1e: medium/high architectural risk because it can mask toolchain correctness problems.

## 7) `Stride.Engine` dependency on shader compiler

### Direct dependency
- `Stride.Engine.csproj` has direct `ProjectReference` to `Stride.Shaders.Compiler.csproj`.

### Source-level coupling
- `Game.cs` imports `Stride.Shaders.Compiler` and calls `EffectCompilerFactory.CreateEffectCompiler(...)` during initialization to set `EffectSystem.Compiler`.
- `EffectCompilerFactory` may create `EffectCompiler` (local), `RemoteEffectCompiler`, or `NullEffectCompiler`, but all these types live in the shader compiler assembly/namespace.

### Classification
- This appears runtime-adjacent but primarily **optional effect compilation path/control plane**, not strictly required for loading precompiled shader bytecode in principle.

### Can it be conditioned out?
- Yes, plausibly, with a narrow property-guarded compile path and project reference condition.
- Expected compile fallout if reference removed without code guards: missing `EffectCompilerFactory` and related types in `Game.cs` and potential rendering-side compiler parameter types.

Recommendation: pursue reversible property-based opt-out rather than deletion.

## 8) Relationship to `Stride.Graphics` and `Stride.Rendering`

- `Stride.Graphics.csproj` references `Stride.Shaders` and `Stride`, but not `Stride.Shaders.Compiler` directly.
- `Stride.Rendering.csproj` only references `Stride.Games`, but rendering source imports `Stride.Shaders.Compiler` in effect-system related files (`EffectSystem.cs`, `RootEffectRenderFeature.cs`, etc.), implying transitive availability expectation from higher-level graph.
- Why M1c could pass while M1e fails:
  - M1c targeted graphics basics slice; `Stride.Engine` admission in M1e adds direct shader compiler dependency and initialization path.
  - M1e script builds `Stride.Engine` slice with transitive closure from that project list.

Inference: failing shader project is tied to compiler/tooling path admitted by `Stride.Engine`, not baseline graphics runtime project admission alone.

## 9) Possible repair paths

### Path A — Restore/fix CppNet dependency
- Likely changes later: ensure hint-path artifact validity for Linux build graph (copy/local ref packaging or modernize reference).
- Risk: medium.
- Pro: preserves legacy shader compiler path.
- Con: cements architecture coupling and may reveal further native/compiler blockers immediately after.

### Path B — Condition out `Stride.Shaders.Compiler` from `Stride.Engine` for Stri‑V Core
- Likely changes later: conditional `ProjectReference` in `Stride.Engine.csproj`; conditional code in `Game.cs` for compiler assignment.
- Risk: medium (localized compile break handling needed).
- Alignment: high with runtime/tooling separation doctrine.

### Path C — Stub/no-op `PreProcessor`
- Minimal code edits possible, but behavior loss is severe for shader source preprocessing.
- Risk: high for hidden breakage.
- Better only as temporary emergency compile unblock, not preferred architectural move.

### Path D — Split shader runtime vs shader compiler projects
- Best long-term boundary (runtime shader/effect model vs source parser/preprocessor/compiler).
- Risk: high and broader refactor; likely beyond M1 repair scope.

### Path E — Defer `Stride.Engine`
- Possible but counter to current admission objective; may delay uncovering real engine runtime boundaries.
- Could be fallback if B proves unexpectedly deep.
